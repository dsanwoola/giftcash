# GiftCash Cloudflare Workers + D1/R2 migration plan

## Current state verified

- `@opennextjs/cloudflare migrate`/`build` and `wrangler deploy --dry-run` were validated in `/tmp/giftcash-cloudflare-spike`.
- The app still selects `firestoreRepo` from `src/lib/data/repo.ts` whenever Firebase public config exists.
- Client Firebase usage remains in:
  - `src/lib/auth/auth-context.tsx` (Auth + profile creation)
  - `src/lib/data/firestore-repo.ts` (Firestore reads/non-ledger writes/realtime)
  - `src/app/admin/bank-alerts/page.tsx` and `src/components/event/event-report-page.tsx` (Firebase ID token for API calls)
- Server Firebase Admin usage remains in:
  - `src/lib/data/server-store.ts`
  - `src/lib/payments/event-payments.ts`
  - `src/lib/payments/reconciliation.ts`
  - `src/app/api/**` payment/admin/report/health routes
- Firebase Storage is initialized but there is no active app upload path using it. `Gift.media` exists, so R2 is only needed when media uploads are implemented or migrated.

## Minimal target architecture

- OpenNext Cloudflare Worker runs the Next app.
- D1 stores all current Firestore collections.
- R2 stores future/user gift media and optional OpenNext incremental cache.
- KV is optional for short-lived/session/cache data; do not use KV as source of truth for ledger/payments because eventual consistency is unsafe for money movement.
- Auth must be decided before removing Firebase:
  - fastest transitional path: keep Firebase Auth only, verify JWTs in Worker via JWKS/REST and move Firestore to D1;
  - full Firebase removal: replace Auth UI/session with Cloudflare-compatible provider (Clerk/Auth.js/custom email OTP) and map provider subject to `profiles.id`.

## D1 schema start

Initial schema: `migrations/0001_d1_initial.sql`.

It maps the existing Firestore documents into relational tables while keeping nested payloads as JSON text where that minimizes churn:

- `profiles`
- `gifts`
- `ledger_entries`
- `withdrawals`
- `group_gifts` + `group_contributions`
- `events` + `event_contributions`
- `payment_intents`
- `bank_alerts`
- `payment_reconciliation_logs`
- `admin_audit_logs`
- `fraud_flags`

## Practical implementation order

1. Bring OpenNext files from the spike into the real repo:
   - `open-next.config.ts`
   - `wrangler.jsonc` with D1 binding, optional R2 bindings, `nodejs_compat`, assets, self-reference service.
   - package scripts/dev deps from spike: `preview`, `deploy`, `upload`, `cf-typegen`, `@opennextjs/cloudflare`, `wrangler`.
2. Create Cloudflare resources:
   - D1 database `giftcash` and bind as `DB`.
   - R2 bucket `giftcash-media` if media uploads are in scope; optionally `giftcash-cache` for OpenNext incremental cache.
   - Worker secrets for Paystack, AgentMail, bank-alert webhook secret, auth/JWT config.
3. Apply `migrations/0001_d1_initial.sql` locally and remotely with Wrangler.
4. Add Cloudflare env access helper, e.g. `src/lib/cloudflare/env.ts`, wrapping `getCloudflareContext().env` or OpenNext equivalent.
5. Add a D1 data access layer:
   - `src/lib/data/d1-mappers.ts` for row/domain conversions.
   - `src/lib/data/d1-repo.ts` implementing `GiftRepo` for client/UI calls via API fetches, not direct D1 from browser.
   - `src/lib/data/d1-server-store.ts` replacing `server-store.ts` Admin SDK transactions with D1 transactions/batches.
6. Move all writes behind API routes. Firestore client writes currently create group gifts/events and update event settings directly; these must become API calls because browsers cannot access D1 securely.
7. Replace realtime Firestore `subscribeEvent`:
   - minimal: poll `/api/events/[slug]` every 2-5 seconds for live screen;
   - better: add Durable Object/WebSocket fanout later.
8. Port payment/bank-alert code from `adminDb()` to D1 queries in one module so route handlers stay thin.
9. Run an export/import migration from Firestore to D1 once credentials/export files are available.
10. Remove `firebase`/`firebase-admin` only after auth replacement/transitional auth verification is complete.

## Blockers / decisions needed

- Auth choice: keep Firebase Auth temporarily or fully replace it. This determines token verification and profile IDs.
- Cloudflare account credentials are needed to create D1/R2 and obtain binding IDs.
- Firestore service account or Firestore export is needed to migrate production data.
- Existing admin users/roles must be identified before import so admin access is preserved.
- Payment secrets are needed in Cloudflare Workers secrets:
  - `PAYSTACK_SECRET_KEY`
  - `PAYSTACK_WEBHOOK_SECRET` if used
  - `BANK_ALERT_WEBHOOK_SECRET`
  - `AGENTMAIL_API_KEY`, `AGENTMAIL_EMAIL`, `AGENTMAIL_WEBHOOK_SECRET` if used
- Realtime event wall must accept polling initially or require Durable Object/WebSocket work.
- D1 transaction semantics should be used for claim/withdrawal/payment confirmation paths before any live money flow is switched.
